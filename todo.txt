# План разработки модульного overhaul-мода для Oxygen Not Included

## 1. Назначение проекта

Разработать мод для Oxygen Not Included, совместимый с:

* базовой игрой;
* DLC Spaced Out!;
* актуальными DLC, если используемые игровые API доступны;
* Steam Workshop;
* Mod Updater.

Мод должен добавлять настраиваемые механики усложнения:

1. Изменение размеров астероидов.
2. Снижение массы ресурсов при добыче.
3. Лучистый перенос тепла в вакууме.
4. Аварийный нагрев повреждаемого провода или моста при штатной перегрузке.
5. Нагрев Solar Panel пропорционально фактической генерации.
6. Тепловой ребаланс топливных генераторов с учётом температуры топлива.
7. Нагрев промышленных зданий в зависимости от мощности.
8. Разрушение капельных жидкостных шлюзов перепадом давления.
9. Усиление Food Poisoning и Slimelung.
10. Изменение скорости получения опыта.
11. Изменение базового потребления калорий.

Исключённые направления, которые не входят в roadmap:

* изменение стартовой температуры биомов — не реализуется, потому что эту нишу уже закрывают отдельные world temperature mods;
* новый сверхпроводящий материал и новые провода — не реализуются, потому что после отказа от сопротивления/потерь проводов у них нет достаточной геймплейной роли;
* электрические потери в проводах, voltage drop, brownout, сопротивление проводов и КПД трансформаторов — не реализуются и не остаются в backlog.

Все механики должны быть независимыми и отключёнными по умолчанию.

# 2. Основные архитектурные принципы

## 2.1. Модульность

Каждая механика оформляется отдельным модулем, реализующим общий контракт:

```csharp
public interface IGameplayModule
{
    string Id { get; }

    bool IsEnabled(ModContext context);

    void Initialize(ModContext context);

    void RegisterPatches(Harmony harmony, ModContext context);

    void OnGameStarted(ModContext context);

    void OnGameLoaded(ModContext context);

    void OnSettingsChanged(
        ModContext context,
        ModSettings previous,
        ModSettings current);

    void Shutdown(ModContext context);
}
```

Точный интерфейс допускается корректировать после исследования текущего API игры.

Основное требование: отключение одного модуля не должно мешать загрузке остальных.

## 2.2. Минимальное вмешательство

Использовать следующий приоритет решений:

1. Официальные данные и конфигурация игры.
2. YAML worldgen.
3. Регистрация новых сущностей через игровые prefab/config API.
4. Postfix/Prefix Harmony-патчи.
5. Transpiler только при отсутствии безопасной альтернативы.
6. Прямое вмешательство в SimMessages или низкоуровневую симуляцию — только после отдельного прототипа.

## 2.3. Fail-safe поведение

Если модуль не смог найти необходимый метод или игровую сущность:

* модуль отключается;
* остальные модули продолжают работать;
* в журнал записывается причина;
* пользователю показывается однократное предупреждение;
* не должно происходить бесконечного выброса исключений каждый тик.

## 2.4. Производительность

Запрещается:

* сканировать всю карту каждый кадр;
* обходить все клетки каждый SimTick;
* использовать LINQ в горячих циклах;
* создавать временные коллекции каждый тик;
* постоянно выполнять reflection;
* логировать каждую клетку или каждый пакет энергии;
* хранить ссылки на уничтоженные объекты.

Для периодических расчётов применять:

* распределение нагрузки между тиками;
* очереди dirty-объектов;
* обработку по событиям;
* spatial buckets;
* кэширование;
* ограниченный бюджет клеток на один кадр;
* периодическое обслуживание вместо полного сканирования.

# 3. Поддерживаемые пресеты

Стандартизировать названия:

```text
Off
VanillaPlus
Hard
Extreme
Insane
Custom
```

Не использовать в коде варианты `Vanila`, `Extrime`, `Ensane`.

## 3.1. Правила пресетов

`Off`:

* все модули отключены;
* vanilla-поведение не изменяется.

`VanillaPlus`:

* умеренные потери и штрафы;
* механики заметны, но не требуют полного перестроения базы.

`Hard`:

* механики существенно влияют на проектирование базы.

`Extreme`:

* требуется специализированная электрическая инфраструктура;
* болезни и ресурсные потери становятся значимыми.

`Insane`:

* максимально жёсткий, но технически проходимый режим;
* значения не должны делать старт математически невозможным.

`Custom`:

* все параметры задаются отдельно;
* применяются ограничения диапазонов.

## 3.2. Предварительная таблица значений

Значения считать стартовыми и вынести в отдельный конфигурационный класс.

| Параметр                    | VanillaPlus |       Hard |   Extreme |         Insane |
| --------------------------- | ----------: | ---------: | --------: | -------------: |
| Масса после добычи          |         50% |        25% |       10% |             1% |
| Опыт                        |        0.8x |       0.6x |     0.35x |          0.15x |
| Калории                     |        1.1x |      1.25x |      1.5x |           2.0x |
| Болезни                     |         10% |        30% |       50% |            80% |
| Solar Panel heat            |   0-5 kDTU/s proportional to actual 0-380 W output |
| Overload conductor heat     |   до 90% температуры плавления материала повреждаемого объекта |
| Fuel generator thermal mode |   профиль мощности/нагрева + учёт температуры топлива |

Не задавать параметры КПД трансформаторов, сопротивления проводов, потерь по длине сети и глобального уменьшения мощности генераторов. Эти идеи исключены из roadmap и не остаются в backlog.

# 5. Конфигурация

## 5.1. Разделение конфигурации

Должно существовать три уровня:

### Глобальная конфигурация

Хранит:

* язык;
* режим логирования;
* диагностику производительности;
* общие UI-настройки;
* последний выбранный пресет для новой игры.

### Конфигурация нового мира

Хранит параметры, применяемые только во время генерации:

* размер астероида;
* режим размера астероида: `Vanilla`, `Half`, `Quarter`;
* флаг experimental worldgen mode;
* результат последней валидации worldgen-настроек;
* включённые worldgen-модули.

### Конфигурация сохранения

Хранит фактические параметры текущей колонии:

* активные runtime-модули;
* коэффициенты;
* версию схемы конфигурации;
* snapshot worldgen-настроек.

После создания мира worldgen-параметры становятся read-only.

## 5.2. Схема конфигурации

```csharp
public sealed class ModSettings
{
    public int SchemaVersion { get; set; }

    public DifficultyPreset Preset { get; set; }

    public WorldSettings World { get; set; }

    public MiningSettings Mining { get; set; }

    public RadiativeHeatSettings RadiativeHeat { get; set; }

    public PowerSettings Power { get; set; }

    public IndustrialHeatSettings IndustrialHeat { get; set; }

    public FluidPressureSettings FluidPressure { get; set; }

    public DiseaseSettings Diseases { get; set; }

    public DuplicantSettings Duplicants { get; set; }

    public DiagnosticsSettings Diagnostics { get; set; }
}
```

## 5.3. Применение пресета

Пресет должен генерировать полный набор настроек:

```csharp
ModSettings PresetFactory.Create(DifficultyPreset preset);
```

После ручного изменения хотя бы одного параметра текущий режим переключается на `Custom`.

Не определять поведение через многочисленные проверки:

```csharp
if (preset == DifficultyPreset.Hard)
```

Игровые модули должны использовать итоговые числовые настройки, а не название пресета.

## 5.4. Валидация

Все пользовательские значения проверяются до применения.

Примерные безопасные диапазоны:

```text
Mining yield:                0.01–1.00
Experience multiplier:      0.05–10.00
Calories multiplier:        0.10–10.00
Disease severity:           0.00–0.95
Overload target ratio:      0.50–0.99
Liquid threshold:           0.01–0.50 клетки
Simulation interval:        0.2–600 секунд
```

Значения NaN, Infinity и отрицательные значения должны отклоняться.

## Логирование

Уровни:

```text

Debug

```

По умолчанию:

```text
Info
```

Debug и Trace включаются вручную.

Для повторяющихся сообщений использовать rate limit:

```text
не более одного одинакового сообщения за 60 секунд
```

## Safe mode

Каждый патч регистрируется через защитную обёртку:

```csharp
PatchResult TryPatch(

    MethodBase target,
    HarmonyMethod prefix,
    HarmonyMethod postfix,
    string moduleId);
```

При ошибке возвращается структурированный результат, а не только исключение.

# 9. Этап 3. Штраф массы при добыче

Это первый игровой модуль, потому что он сравнительно изолирован и удобен для проверки архитектуры.

## Поведение

После разрушения доступной для копания клетки:

```text
полученная масса = исходная добываемая масса × mining yield
```

Поддерживаемые варианты:

```text
Default
50%
25%
10%
1%
Custom
```

`Default` означает vanilla-поведение, а не обязательно 100% исходной массы клетки.

Важно сначала определить, на каком этапе игра уже применяет стандартный коэффициент потери массы. Наш коэффициент должен применяться один раз и не дублировать vanilla-штраф.

## Исключения

Не изменять:

* нейтрониум;
* недоступные для копания элементы;
* клетки, уничтоженные не копанием, если это не включено отдельно;
* ресурсы, созданные зданиями;
* содержимое построек;
* элементы, выпавшие из существ;
* ресурсы из ракет и POI.

## Уникальные ресурсы

Создать отдельный реестр ресурсов, которые становятся практически невосполнимыми при низком коэффициенте добычи:

```csharp
public interface IScarceResourceRecipeProvider
{
    void RegisterRecipes();
}
```

На первом этапе не добавлять рецепты автоматически для всех редких ресурсов.

Сформировать документ:

```text
docs/scarce-resource-balance.md
```

В нём перечислить:

* ресурс;
* источник;
* риск исчерпания;
* предлагаемый рецепт;
* технологический уровень;
* стоимость;
* побочные продукты.

Абиссалит должен получить дорогой поздний рецепт. Нейтроний не должен становиться доступным.

## Тесты

Проверить:

* твёрдую клетку;
* руду;
* лёд;
* абиссалит;
* разрушение клеток разными инструментами;
* сохранение/загрузку;
* разные пресеты;
* отсутствие двойного применения коэффициента.

# 10. Этап 4. Опыт и калории

## Опыт

Изменяется только скорость получения опыта, необходимого для skill point.

```text
effective XP = vanilla XP × experience multiplier
```

Не менять:

* стоимость навыков;
* требования морали;
* атрибутный опыт;
* эффективность Skill Scrubber.

Перед реализацией проверить, разделяет ли игра опыт атрибутов и опыт skill points. Патч должен применяться только к нужному типу опыта.

## Калории

Изменяется базовая скорость расхода калорий:

```text
effective calorie consumption =
    vanilla consumption × calorie multiplier
```

Не менять:

* калорийность еды;
* запас желудка;
* производство еды;
* индивидуальные traits, если они уже применяются игрой.

Множитель должен применяться поверх vanilla-модификаторов, а не заменять их.

## Тесты

* работа;
* сон;
* бездействие;
* костюм;
* голод;
* сохранение/загрузка;
* дубликант с модификатором метаболизма.

# 11. Этап 5. Болезни

## Food Poisoning

При активной болезни уменьшать общую продуктивность:

```text
VanillaPlus: -10%
Hard:        -30%
Extreme:     -50%
Insane:      -80%
```

Не следует патчить каждый рабочий chore отдельно.

Предпочтительно использовать общий modifier:

* скорость выполнения работы;
* скорость действий;
* либо существующую систему AttributeModifier.

Нужно выбрать один механизм, который не уменьшает параметр несколько раз.

## Slimelung

При активной болезни:

* уменьшать мораль;
* увеличивать скорость накопления стресса.

Пропорции:

```text
10%, 30%, 50%, 80%
```

Для морали процентное снижение может быть плохо совместимо с дискретной шкалой ONI. Допускается преобразовать уровни в фиксированные значения после изучения реальных диапазонов.

Пример:

```text
VanillaPlus: -1 morale
Hard:        -3 morale
Extreme:     -5 morale
Insane:      -8 morale
```

Стресс считать множителем к существующему приросту, а не постоянным безусловным приростом, если это возможно.

## Ограничения

* модификаторы удаляются после выздоровления;
* после загрузки сохранения нет дубликатов;
* повторное заражение не суммирует одинаковые эффекты;
* отключение модуля очищает runtime-компоненты, насколько это безопасно.

# 12. Этап 6. Тепловой ребаланс генераторов и Solar Panel

Этот раздел переопределяет старую идею `КПД генераторов`. В активной версии 0.4 не реализуется глобальный множитель КПД, который просто уменьшает всю выдаваемую мощность. Вместо этого используется профильная матрица топливных генераторов, тепловой учёт входящего топлива и отдельная модель Solar Panel.

## 12.1. Топливные генераторы

Блок генераторов должен сохранять следующие инварианты:

* расход топлива не изменяется;
* тип топлива не изменяется;
* масса продуктов не изменяется;
* тип продуктов не изменяется;
* новые входы и выходы не добавляются;
* температура входящего топлива учитывается;
* дополнительная тепловая энергия не уничтожается;
* жидкие продукты не должны штатно переходить в другую фазу внутри исправного генератора;
* остаточная энергия передаётся газовому выходу и корпусу;
* Steam Turbine использует отдельную штатную модель;
* Solar Panel не обрабатывается как топливный генератор;
* Tidal Generator и Dischargers не обрабатываются как топливные генераторы.

Стартовая матрица топливных генераторов:

| Генератор             | Мощность                  | Базовый нагрев корпуса | Учёт температуры топлива |
| --------------------- | ------------------------: | ---------------------: | ------------------------ |
| Manual Generator      |                     400 W |               1 kDTU/s | нет                      |
| Wood Burner           |                     300 W |              20 kDTU/s | да                       |
| Coal Generator        |                     600 W |              40 kDTU/s | да                       |
| Peat Burner           | 480 W или runtime vanilla |            22.5 kDTU/s | да                       |
| Hydrogen Generator    |                     800 W |              32 kDTU/s | да, 100% в корпус        |
| Natural Gas Generator |                     800 W |              20 kDTU/s | да                       |
| Petroleum Generator   |                    2000 W |              50 kDTU/s | да                       |
| Steam Turbine         |                   vanilla |        vanilla dynamic | отдельная модель         |
| Tidal Generator       |                   vanilla |        vanilla/runtime | нет                      |
| Compact Discharger    |                      60 W |                vanilla | нет                      |
| Large Discharger      |                     480 W |                vanilla | нет                      |

## 12.2. Runtime-модель генераторов

Для топливных генераторов:

1. Перед штатным расходом топлива захватить массу, температуру и SHC входящего топлива из `Storage`.
2. После штатного тика определить фактически потреблённую массу.
3. Рассчитать surplus тепла:

```text
FuelSurplusDtu =
    ConsumedMassKg × 1000 × FuelSHC × max(0, FuelTemperatureK - BodyTemperatureK)
```

4. Газовые продукты получают температуру не ниже захваченной температуры топлива.
5. Жидкие продукты получают температуру не выше `Element.highTemp - 1 K`, чтобы вода и загрязнённая вода не превращались в пар внутри исправного генератора.
6. Энергия, уже назначенная продуктам, вычитается из surplus перед добавлением остатка в корпус.
7. Нативное operating/exhaust heat для профилированных генераторов подавляется, чтобы не было двойного нагрева.
8. Тепло корпуса добавляется через штатный thermal API игры, а не прямой записью температуры.
9. Значения в свойствах здания и hover tooltip должны соответствовать runtime-поведению.

## 12.3. Solar Panel

Solar Panel является преобразователем света, а не топливным генератором.

Требования:

* heat range: `0-5 kDTU/s`;
* power range: `0-380 W`;
* нагрев пропорционален фактической текущей мощности;
* источник света модом не сканируется;
* sunlight, Shine Bugs и другие источники света работают одинаково, если игра уже отразила их во `CurrentWattage`;
* static descriptor показывает максимум `5 kDTU/s`;
* hover tooltip показывает текущий нагрев по фактической мощности.

Формула:

```text
HeatDtuPerSecond =
    clamp(CurrentWattage, 0, 380) / 380 × 5000
```

## 12.4. Тесты генераторов и Solar Panel

Для каждого профилированного генератора проверить:

* заявленную мощность;
* фактическую мощность;
* расход топлива;
* температуру корпуса при холодном и горячем топливе;
* температуру газовых продуктов при горячем топливе;
* безопасный cap жидких продуктов ниже фазового перехода;
* соответствие `properties` и hover tooltip;
* автоматизацию;
* сохранение и перезагрузку;
* отключение модуля.

Для Solar Panel проверить:

* 0 W -> 0 kDTU/s;
* 95 W -> 1.25 kDTU/s;
* 190 W -> 2.5 kDTU/s;
* 285 W -> 3.75 kDTU/s;
* 380 W -> 5 kDTU/s;
* значения выше 380 W clamp к 5 kDTU/s;
* static descriptor и hover tooltip не противоречат друг другу.

# 13. Этап 7. Аварийный нагрев проводов при перегрузке

Этот раздел переопределяет старую идею электрических потерь по длине проводов. В активной версии 0.4 не моделируются сопротивление сети, voltage drop, brownout потребителей, постоянный резистивный нагрев исправного провода и потери мощности по длине.

## 13.1. Активная модель

Когда ONI выбирает конкретный провод, мост или совместимый объект для штатного `overload damage`, мод добавляет тепловой импульс именно этому повреждаемому объекту.

Инварианты:

* штатный overload damage сохраняется;
* нагревается только объект, которому игра нанесла damage;
* соседние провода не получают прямой импульс;
* объект нагревается до 90% температуры плавления строительного материала;
* температура не задаётся напрямую;
* энергия рассчитывается через массу и SHC;
* фазовый переход не форсируется модом;
* один damage event не должен дать несколько тепловых импульсов.

Расчёт:

```text
TargetTemperatureK = MaterialMeltingTemperatureK × 0.90
RequiredEnergyDtu =
    max(0, TargetTemperatureK - CurrentTemperatureK)
    × BuildingMassGrams
    × MaterialSHC
```

Если текущая температура уже выше target, мод не охлаждает объект.

## 13.2. Объекты

Поддержать:

* Wire;
* Conductive Wire;
* Heavy-Watt Wire;
* Heavy-Watt Conductive Wire;
* Wire Bridge;
* Conductive Wire Bridge;
* Heavy-Watt Joint Plate, если штатная игра выбирает её для overload damage;
* новые провода мода;
* будущие DLC-провода, если они используют совместимый damage path.

Для мостов tooltip должен показывать те же параметры, что и для проводов.

## 13.3. Fail-safe

Если не удалось определить material, mass, SHC, melting point или thermal sim handle:

* штатный damage сохраняется;
* дополнительный нагрев пропускается;
* пишется rate-limited warning;
* остальные модули не отключаются.

## 13.4. Tooltip

Для проводов и мостов отображать:

```text
Material
Melting point
Overload heat target
Last overload heat
```

Не отображать в версии 0.4:

```text
Wire resistance
Voltage drop
Length loss
Delivered power
Transformer loss
```

# 14. Исключённые электрические направления

Следующие идеи исключены из roadmap и не должны реализовываться в рамках этого проекта:

* КПД трансформаторов;
* потери в проводах по длине;
* сопротивление проводов по материалу и температуре;
* voltage drop;
* brownout потребителей;
* деградация скорости производства из-за недостающей мощности;
* постоянный резистивный нагрев исправных проводов от нагрузки;
* глобальное уменьшение количества генерируемой мощности через единый efficiency multiplier;
* новый сверхпроводящий материал;
* новые обычные или high-watt провода.

Причины:

* ONI не даёт достаточно прозрачной модели направленного потока энергии от источника к потребителю для аккуратной реализации физической сети без собственной сети поверх игры;
* после отказа от сопротивления/потерь новые провода и сверхпроводник теряют основную геймплейную роль;
* поддержка отдельной электрической модели увеличит риск конфликтов с другими модами и усложнит баланс без достаточной ценности.

Разрешённая электрическая механика проекта: только аварийный нагрев объекта, которому vanilla overload damage уже нанесла повреждение.

# 16. Этап 10. Нагрев промышленных зданий

## Область

Только промышленные здания.

Нужен реестр:

```csharp
IndustrialBuildingRegistry
```

Автоматическое обнаружение возможно по:

* категории;
* компонентам;
* типу complex fabricator;

```text
additionalHeat =
    currentPowerConsumption
    × industrialHeatFactor
```

Нагрев добавляется только во время фактической работы.

```text
vanilla heat + mod additional heat
```

Если настройка в будущем будет называться multiplier, тогда:

```text
vanilla heat × multiplier
```

Не смешивать две модели.

# 18. Версия 0.7. Размер астероидов

Это следующий активный этап после стабилизации 0.4.7.

Модуль относится к worldgen и считается высокорисковым: ошибка может ломать создание мира, обязательные структуры, телепорты, story traits, космические destinations или сохранение.

## 18.1. Цель

Добавить экспериментальные уменьшенные варианты астероидов:

```text
Vanilla
Half
Quarter
```

Для Spaced Out цель формулируется как уменьшение отдельных стартовых/колонизируемых астероидов, а не как полное физическое масштабирование всей star map.

Не делать в 0.7:

* изменение температуры биомов;
* новые биомы;
* новые story traits;
* изменение правил ракет;
* изменение geyser output;
* изменение количества ресурсов после генерации;
* runtime-сжатие уже созданного мира.

## 18.2. Принципы реализации

Приоритет решений:

1. Data/YAML worldgen overrides.
2. Добавление новых world/cluster presets рядом с vanilla, без изменения vanilla presets.
3. Минимальные Harmony patches только для выбора новых presets или безопасной регистрации данных.
4. Runtime worldgen patches только если YAML/data path недостаточен.

Нельзя:

* менять уже созданные сохранения;
* переписывать vanilla YAML in-place;
* ломать запуск игры при отсутствии DLC;
* сканировать или перестраивать мир после генерации;
* полагаться на один seed как доказательство корректности.

## 18.3. Исследовательский этап

Перед реализацией найти в текущих игровых файлах и DLL:

* где лежат cluster/world YAML;
* как игра выбирает cluster layout;
* как задаются `worldsize`, borders и стартовая область;
* как размещаются biome templates;
* как размещаются geysers, POI, teleporters, Temporal Tear и story-critical объекты;
* какие отличия есть между base game и Spaced Out;
* как мод может добавить новый world preset без перезаписи vanilla данных;
* где безопасно валидировать выбранный preset до старта генерации.

Результат исследования зафиксировать в:

```text
docs/v0.7-worldgen-asteroid-size-audit.md
```

## 18.4. YAML-only прототип

Первый прототип должен быть data-only, если это возможно.

Проверить:

* width;
* height;
* starting location;
* world border;
* biome counts;
* biome template bounds;
* POI placement;
* geysers;
* teleporters;
* Temporal Tear;
* rocket destinations;
* DLC-specific structures.

## 18.5. Валидатор генерации

До запуска генерации или сразу после сборки worldgen data проверять:

* стартовый биом помещается в карту;
* стартовая область безопасна для появления duplicants;
* обязательные структуры помещаются в границы;
* нет координат за пределами мира;
* количество geysers/POI не становится невозможным для размера карты;
* teleporters и paired structures не теряют пару;
* story-critical structures присутствуют или режим явно помечен experimental;
* DLC-only элементы не используются без соответствующего DLC.

При провале валидации:

* не начинать генерацию, если ошибка обнаружена до неё;
* показывать понятное предупреждение;
* писать diagnostics;
* не менять vanilla presets.

## 18.6. Режимы размера

`Half`:

* основной целевой режим 0.7;
* должен пройти широкий набор seed-tests;
* должен быть пригоден для обычного прохождения, если vanilla worldgen constraints позволяют.

`Quarter`:

* experimental;
* допускается как режим для стресс-теста;
* может отключаться для отдельных clusters/DLC, если обязательные структуры не помещаются стабильно.

## 18.7. UI и конфигурация

До полноценного UI допустимо использовать config.

Настройки:

```json
{
  "World": {
    "Enabled": true,
    "AsteroidSize": "Half"
  }
}
```

Требования:

* настройка применяется только к новым мирам;
* для существующего сохранения настройка read-only и не пытается сжать карту;
* vanilla остаётся значением по умолчанию;
* experimental режимы должны явно предупреждать о риске worldgen failure.

## 18.8. Обязательные тесты

Для каждого размера:

* не менее 20-50 генераций с разными seed на этапе прототипа;
* base game;
* Spaced Out;
* разные стартовые астероиды;
* все установленные DLC, если их worldgen data доступна;
* проверка отсутствия worldgen exception;
* появление обязательных объектов;
* создание колонии;
* сохранение и повторная загрузка;
* отключение модуля и генерация vanilla мира.

Создать журнал результатов:

```text
seed
world preset
asteroid size mode
DLC
result
generation time
exception
missing entities
validation errors
```

## 18.9. Критерии готовности 0.7

Версия 0.7 готова, если:

1. Vanilla generation не меняется при выключенном модуле.
2. Half mode стабильно создаёт миры на выбранном наборе clusters/seeds.
3. Quarter mode либо стабилен, либо явно помечен experimental и ограничен по clusters.
4. Нет runtime-сжатия существующих миров.
5. Нет перезаписи vanilla worldgen файлов.
6. Есть diagnostics для неудачной генерации.
7. Документированы ограничения для base game, Spaced Out и DLC.

# 19. Этап 13. Лучистый перенос тепла в вакууме

Это не радиация в смысле болезней или DLC Radiation. В коде модуль называть:

```text
RadiativeHeat
```

Не использовать `Radiation`, чтобы не смешивать две разные системы.

## 19.1. Модель

Горячий объект может передавать тепло другим видимым объектам через вакуум.

Базовая физическая формула:

```text
P = ε × σ × A × (Thot⁴ - Tcold⁴)
```

Но прямое применение реальных единиц может быть непригодно для баланса ONI.

Использовать настраиваемую нормализованную модель:

```text
Pradiated =
    radiationFactor
    × emissivity
    × effectiveArea
    × normalizedTemperatureDifference
```

Дополнительно предусмотреть физический режим после стабилизации.

## 19.2. Источники

Не считать источником каждую горячую клетку карты.

Первоначально учитывать только:

* здания;
* твёрдые клетки выше температурного порога;
* жидкие клетки выше порога;
* объекты с зарегистрированным thermal emitter component.

## 19.3. Условия передачи

Передача происходит только если:

* между источником и целью вакуум либо допустимая оптически прозрачная среда;
* нет непрозрачной твёрдой преграды;
* цель находится в радиусе;
* разница температур выше порога;
* источник имеет достаточную температуру;
* модуль включён.

## 19.4. Line of sight

Не выполнять raycast от каждого источника ко всем клеткам.

Использовать:

* spatial partition;
* ограниченный радиус;
* фиксированное число лучей;
* кэш видимости;
* invalidation при изменении клеток;
* очередь источников;
* ограниченный бюджет расчётов.

## 19.5. Сохранение энергии

Тепло, добавленное целям, должно вычитаться у источника:

```text
ΣQreceived ≤ QremovedFromSource
```

Не создавать энергию из ничего.

Если несколько целей:

* распределять энергию по видимому углу;
* расстоянию;
* эффективной площади;
* температурной разнице.

## 19.6. Частота

Начальный интервал:

```text
1 раз в секунду игрового времени
```

Для большого числа источников распределять расчёт по кадрам.

## 19.7. Ограничения первой версии

Не реализовывать сразу:

* отражение;
* многократное переизлучение в одном цикле;
* спектры;
* длины волн;
* полупрозрачные материалы;
* сложную оптику;
* солнечные лучи между астероидами.

# 20. Этап 14. Давление и капельные шлюзы

Это наиболее рискованный модуль после worldgen.

## 20.1. Цель

Если малая масса жидкости разделяет области с разным давлением, газ должен иметь возможность просачиваться или вытеснять жидкость.

Примеры:

```text
Газ 10 кг → тонкий слой жидкости → вакуум
```

```text
Природный газ 10 кг → нефтяная капля → кислород 1 кг
```

## 20.2. Порог

Требование:

```text
liquid mass < 10% полной клетки
```

Проблема: “полная клетка” зависит от плотности и типа жидкости.

Поэтому вычислять:

```text
fillRatio = liquidMass / referenceFullCellMass(liquidElement)
```

Порог:

```text
fillRatio < 0.10
```

Для каждого элемента использовать реальную или игровую эталонную массу полной клетки.

## 20.3. Перепад давления

```text
ΔP = pressureHigh - pressureLow
```

Просачивание происходит только если:

```text
ΔP > minimumPressureDifference
```

Скорость:

```text
transferMass =
    permeabilityFactor
    × ΔP
    × (1 - fillRatio)
```

Ограничить:

* доступной массой газа;
* максимальной массой за цикл;
* вместимостью принимающей клетки;
* минимальным остатком;
* безопасными пределами SimMessages.

## 20.4. Направление

Проверять четыре ортогональных направления.

Диагональное просачивание в первой версии не реализовывать.

## 20.5. Что происходит с жидкостью

Первая версия:

* газ переносится через клетку;
* жидкость не уничтожается;
* при достаточном перепаде допускается небольшое перемещение жидкости в сторону низкого давления, если игровая симуляция позволяет это безопасно.

Не телепортировать жидкость через твёрдые клетки.

## 20.6. Выбор клеток

Не сканировать всю карту каждый цикл.

Поддерживать реестр потенциальных barrier cells.

Клетка попадает в реестр, если:

* содержит жидкость;
* fill ratio ниже порога;
* имеет газ или вакуум по обе стороны;
* нет твёрдого блока.

Клетка удаляется, если:

* стала полной;
* высохла;
* была застроена;
* перестала разделять газовые области.

Периодически проводить ограниченную фоновую перепроверку для восстановления пропущенных событий.

## 20.7. Интервал

Требование — один раз в цикл.

Реализовать обработку один раз за игровой цикл, но распределять список клеток на несколько кадров в начале/конце цикла, чтобы избежать spike.

Дополнительно разрешить диагностический интервал в настройках разработчика.

## 20.8. Исключения

Не учитывать:

* бутылки;
* жидкость внутри труб;
* жидкость в зданиях;
* содержимое резервуаров;
* клетки за твёрдой стеной;
* недоступные клетки;
* элементы с некорректной массой.

# 22. Совместимость

## Обязательное правило

Мод считается совместимым с модами, которые не патчат те же механики, только после тестирования.

## Категории конфликтов

```text
World generation mods
Power rebalance mods
Generator rebalance mods
Thermal simulation mods
Liquid physics mods
Disease overhaul mods
Duplicant metabolism mods
Experience mods
```

## Детектор

При загрузке:

1. Получить список активных модов.
2. Сравнить с known conflicts.
3. Показать предупреждение.
4. Не запрещать запуск без крайней необходимости.
5. При необходимости отключить только конфликтующий модуль.

## Safe mode

Если обнаружен конфликтующий мод:

```text
Module disabled because another known mod modifies the same gameplay area.
```

Предусмотреть пользовательский override:

```text
Force enable
```

С предупреждением, что поддержка при этом не гарантируется.

## Mod Updater

Не встраивать код Mod Updater в собственную сборку.

Обеспечить нормальную структуру версии и Workshop-пакета, чтобы Mod Updater мог корректно определить обновление. Современные репозитории ONI-модов по-прежнему рекомендуют Mod Updater из-за случаев, когда Steam выдаёт устаревшие файлы.

# 24. Тестовая стратегия

## 24.1. Unit-тесты вне игры

Покрыть:

* preset generation;
* validation;
* power generation heat calculators;
* solar generation heat calculators;
* overload conductor heat calculators;
* pressure transfer;
* temperature conversion;
* mining coefficients;
* config migration.

## 24.2. Integration-тесты в игре

Создать отдельные debug-команды или sandbox-карты:

```text
Mining Test
Generator Test
Solar Panel Test
Overload Wire Test
Industrial Heat Test
Radiative Heat Test
Liquid Lock Test
Disease Test
Duplicant XP Test
Worldgen Test
```

## 24.3. Инварианты

### Перегрузка проводов

```text
vanilla overload damage is preserved
only damaged conductor receives emergency heat
target temperature < material phase transition
one overload event <= one heat impulse
```

### Теплоизлучение

```text
heat received <= heat removed
```

### Газ

```text
transferred mass <= source mass
mass is not duplicated
```

### Добыча

```text
output mass <= vanilla output mass
```

## 24.4. Длительные тесты

Минимум:

* 100 циклов тестовой базы;
* 500 циклов ускоренной симуляции;
* поздняя база;
* полностью исследованная карта;
* несколько астероидов;
* сохранение/загрузка;
* переключение runtime-модулей;
* перезапуск игры;
* включённый и выключенный diagnostics mode.

# 26. Правила работы Codex

Codex должен работать по одному этапу за раз.

Перед каждым этапом:

1. Изучить существующий код.
2. Проверить актуальные игровые сигнатуры.
3. Составить краткий implementation note.
4. Перечислить изменяемые файлы.
5. Указать риски.
6. Реализовать минимальный вертикальный срез.
7. Собрать проект.
8. Устранить предупреждения.
9. Подготовить ручной тест.
10. Обновить документацию.

Codex запрещается:

* реализовывать несколько крупных модулей одновременно;
* оставлять пустые catch-блоки;
* подавлять все исключения;
* использовать reflection в горячем цикле;
* применять Harmony `TargetMethod` без проверки результата;
* копировать старые сигнатуры из чужих модов без проверки;
* сохранять runtime-ссылки в JSON;
* менять worldgen-настройки уже созданного мира;
* внедрять низкоуровневые патчи без прототипа;
* публиковать Workshop-сборку автоматически;
* добавлять сторонний код без проверки лицензии.

# 27. Definition of Done для каждого модуля

Модуль считается завершённым, если:

* имеет отдельный toggle;
* выключен по умолчанию;
* поддерживается пресетами;
* имеет Custom-настройки;
* значения валидируются;
* настройки сохраняются;
* модуль не запускает обработку в состоянии Off;
* ошибки ограничены rate limit;
* есть unit-тесты формул;
* есть ручной integration test;
* есть performance metrics;
* сохранение загружается;
* русская и английская локализации заполнены;
* documented known conflicts;
* нет постоянных исключений в журнале;
* измерено влияние на скорость симуляции.
